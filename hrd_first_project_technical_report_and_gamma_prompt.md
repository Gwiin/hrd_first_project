# hrd_first_project 최종 기술서 및 감마(Gamma)용 상세 PPT 프롬프트

## 1. 최종 기술서

# MQTT 기반 스마트 하우스 분산 제어 시스템 기술서

## 1. 프로젝트 개요

### 1.1 프로젝트명
MQTT 기반 스마트 하우스 분산 제어 시스템

### 1.2 프로젝트 목적
본 프로젝트의 목적은 스마트 하우스 환경에서 조명과 창문을 자동 또는 수동으로 제어할 수 있는 분산형 시스템을 구현하는 것이다. 단순히 한 보드에서 센서를 읽고 LED를 켜는 수준이 아니라, 센서 노드, 중앙관리 노드, 액추에이터 노드, 웹 관제 콘솔을 분리하여 MQTT 브로커를 통해 상호작용하도록 설계하였다. 이를 통해 확장성이 높고 역할 분리가 명확한 구조를 만들고자 하였다.

### 1.3 개발 배경
스마트 하우스 시스템은 환경 감지, 제어 정책, 장치 구동, 사용자 인터페이스가 동시에 필요하다. 하지만 모든 기능을 하나의 보드에 집중시키면 구조가 복잡해지고 유지보수가 어려워진다. 본 프로젝트는 이런 문제를 해결하기 위해 각 기능을 별도의 노드로 분리하고, MQTT를 통해 느슨하게 연결하는 구조를 선택하였다. 이 구조는 추후 센서나 액추에이터를 추가하더라도 전체 시스템을 크게 수정하지 않고 확장할 수 있다는 장점이 있다.

## 2. 시스템 아키텍처

본 시스템은 크게 네 가지 구성요소로 이루어진다.

- `node_a`: 센서 노드
- `node_b`: 액추에이터 노드
- `node_c`: 중앙관리 노드
- `web_console`: 웹 관제 인터페이스

각 노드는 MQTT 브로커를 중심으로 통신한다.  
`node_a`는 환경 데이터를 측정하여 발행하고, `node_c`는 이를 바탕으로 자동 제어 정책을 수행한다. `node_b`는 실제 하드웨어를 제어하며, `web_console`은 시스템 상태를 시각화하고 사용자의 수동 입력을 받아 명령을 발행한다.

### 2.1 계층 관점 구조
시스템을 계층적으로 보면 다음과 같이 나눌 수 있다.

#### 현장 계층(Field Nodes)
- `node_a`: CDS와 DHT를 통해 환경값 측정
- `node_b`: WS2812 조명과 듀얼 서보 창문 구동

#### 중앙 관리 계층
- `node_c`: 환경 데이터를 기반으로 AUTO / MANUAL 모드 관리 및 제어 정책 수행
- MQTT Broker: 모든 메시지 전달의 중심

#### 사용자 계층
- `web_console`: 전체 상태 확인 및 명령 전송

## 3. 하드웨어 구성

### 3.1 node_a 하드웨어
`node_a`는 환경 센서 노드로 동작한다.

- DHT data pin: `GP15`
- CDS ADC pin: `GP27`

DHT 센서는 온도와 습도를 측정하고, CDS 센서는 조도 값을 ADC로 읽는다. 측정된 데이터는 MQTT를 통해 `house/env`로 발행된다.

### 3.2 node_b 하드웨어
`node_b`는 액추에이터 노드로 동작한다.

- SG90 servo primary: `GP14`
- SG90 servo secondary: `GP15`
- WS2812 data pin: `GP16`

최신 리포지토리 기준으로 창문 제어는 단일 서보가 아니라 **듀얼 서보 카세먼트 구조**로 구현되어 있다. 하나의 MQTT 명령으로 두 서보가 동시에 움직이며, 두 번째 서보는 반대 방향 펄스폭으로 동작하도록 설정되어 양쪽 창짝을 함께 열고 닫을 수 있다. 또한 두 서보를 동시에 움직일 때 전류 피크를 줄이기 위해 펌웨어에서 `secondary -> primary` 순으로 약간의 시간차를 두고 구동한다.

조명은 일반 GPIO LED가 아니라 `WS2812 RGB Strip` 8픽셀 전체를 켜고 끄는 방식으로 구현되어 있다. `lamp ON`이면 전체가 흰색으로 점등되고, `lamp OFF`이면 전체 소등된다.

### 3.3 node_c 하드웨어
`node_c`는 중앙관리용 Raspberry Pi Pico W 기반 제어 노드이다.  
센서와 액추에이터를 직접 연결하기보다, MQTT 브로커와 통신하며 제어 정책과 상태 관리를 담당한다.

### 3.4 전원 및 배선 고려사항
듀얼 서보는 Pico 보드의 핀 전원만으로는 부족할 수 있으므로 외부 5V 전원을 사용하는 것이 권장된다. 또한 두 서보와 Pico의 GND는 반드시 공통으로 묶어야 한다. WS2812 역시 데이터 라인뿐 아니라 VCC와 GND 연결이 안정적이어야 한다.

## 4. 소프트웨어 구성

### 4.1 node_a: 센서 노드
`node_a`는 CDS 조도 센서와 DHT 온습도 센서를 읽고, 그 결과를 MQTT로 발행하는 역할을 한다.

주요 기능은 다음과 같다.

- Wi-Fi 연결
- MQTT 브로커 접속
- DHT 센서 데이터 읽기
- CDS 조도값 읽기
- 조도값 정규화
- `house/env` 발행
- `house/heartbeat/nodeA` 발행

조도 ADC raw 값은 원래 `0~4095` 범위이지만, `node_c`의 임계값과 바로 연동되도록 `0~400` 범위로 정규화하여 발행한다. 이 때문에 중앙 제어 노드는 별도의 변환 없이 조명 ON/OFF 임계값을 적용할 수 있다.

### 4.2 node_b: 액추에이터 노드
`node_b`는 MQTT 명령을 받아 실제 액추에이터를 제어한다.

주요 기능은 다음과 같다.

- `house/cmd/light` 구독
- `house/cmd/window` 구독
- WS2812 8픽셀 전체 ON/OFF 제어
- 듀얼 서보 기반 창문 OPEN/CLOSE
- `house/status/nodeB` 발행
- `house/heartbeat/nodeB` 발행

조명 제어는 PIO 기반 WS2812 제어를 사용하므로 단순 GPIO 방식과 다르다. 창문 제어는 두 서보를 함께 움직이되, 방향과 타이밍을 조정하여 카세먼트 구조를 구현하였다.

### 4.3 node_c: 중앙관리 노드
`node_c`는 본 프로젝트의 핵심 제어기이다.

주요 기능은 다음과 같다.

- `AUTO` / `MANUAL` 모드 관리
- `house/env` payload 파싱
- 임계값 기반 자동 제어
- MQTT 수동 명령 처리
- UART 수동 명령 처리
- `nodeA`, `nodeB` timeout 감시
- `house/status/nodeC` 발행
- `house/heartbeat/nodeC` 발행

자동 제어 기준은 다음과 같다.

- 조명 ON: `light < 280`
- 조명 OFF: `light > 320`
- 창문 OPEN: `temp > 28.0` 또는 `humidity > 70.0`
- 창문 CLOSE: `temp <= 27.0` 그리고 `humidity <= 65.0`

AUTO 모드에서는 수동 `light/window` MQTT 명령을 무시한다. MANUAL 모드에서는 웹 콘솔이나 UART를 통해 수동 제어가 가능하다.

### 4.4 web_console: 웹 관제 인터페이스
`web_console`은 Flask + MQTT 기반 웹 애플리케이션이다.

주요 기능은 다음과 같다.

- `house/#` 전체 구독
- 환경 데이터 표시
- `nodeB`, `nodeC` 상태 표시
- `nodeA`, `nodeB`, `nodeC` heartbeat 표시
- 최근 MQTT 송수신 로그 표시
- 웹 버튼을 통한 명령 발행

웹 콘솔은 `house/mode`, `house/cmd/light`, `house/cmd/window`를 발행할 수 있으며, 이로써 사용자는 웹에서 직접 AUTO / MANUAL 전환과 조명 / 창문 수동 제어를 수행할 수 있다.

## 5. MQTT 통신 구조

### 5.1 환경 데이터 토픽
- `house/env`

이 토픽은 `node_a`가 발행하며, `node_c`가 이를 구독해 자동 제어 판단의 기준으로 사용한다. 웹 콘솔도 이 값을 표시하여 사용자에게 현재 환경을 보여준다.

### 5.2 제어 명령 토픽
- `house/mode`
- `house/cmd/light`
- `house/cmd/window`

`house/mode`는 AUTO / MANUAL 전환에 사용된다. `web_console`이 발행하고 `node_c`가 이를 구독한다.

`house/cmd/light`와 `house/cmd/window`는 조명 및 창문 제어 명령 토픽이다. AUTO 모드에서는 `node_c`가 발행하고, MANUAL 모드에서는 웹 콘솔이 발행한다. `node_b`는 이를 구독하여 실제 장치를 제어한다. `node_c`도 이 명령을 수신하여 상태를 반영한다.

### 5.3 상태 보고 토픽
- `house/status/nodeB`
- `house/status/nodeC`

`node_b`는 액추에이터 반영 결과를 상태 스냅샷으로 발행하고, `node_c`는 현재 모드와 환경값, 조명 및 창문 상태를 종합한 상태 정보를 발행한다. 웹 콘솔은 이 정보를 통해 현재 시스템 상태를 표시한다.

### 5.4 heartbeat 토픽
- `house/heartbeat/nodeA`
- `house/heartbeat/nodeB`
- `house/heartbeat/nodeC`

각 노드는 주기적으로 heartbeat를 발행하여 살아 있음을 알린다. 웹 콘솔은 이를 이용해 online / offline 상태를 표시한다. `node_c`는 `nodeA`, `nodeB` timeout 경고 로직도 포함하고 있다.

## 6. 제어 로직

### 6.1 AUTO 모드
AUTO 모드에서는 `node_c`가 자동 제어 정책을 수행한다.

1. `node_a`가 조도, 온도, 습도를 측정한다.
2. 측정값을 `house/env`로 발행한다.
3. `node_c`가 환경값을 수신한다.
4. `node_c`는 현재 모드가 AUTO인지 확인한다.
5. 조도값을 기준으로 조명 ON/OFF를 판단한다.
6. 온도와 습도를 기준으로 창문 OPEN/CLOSE를 판단한다.
7. 필요한 경우 `house/cmd/light`, `house/cmd/window`를 발행한다.
8. `node_b`가 명령을 수신하여 WS2812 조명과 듀얼 서보 창문을 실제 제어한다.
9. `node_b`와 `node_c`는 상태를 발행한다.
10. `web_console`이 상태와 heartbeat를 반영한다.

### 6.2 MANUAL 모드
MANUAL 모드에서는 사용자가 직접 시스템을 제어한다.

1. 사용자가 웹 콘솔에서 `Mode Manual`을 선택한다.
2. 웹 콘솔이 `house/mode = MANUAL`을 발행한다.
3. `node_c`가 MANUAL 모드로 전환된다.
4. 사용자가 조명 또는 창문 버튼을 누른다.
5. 웹 콘솔이 `house/cmd/light` 또는 `house/cmd/window`를 발행한다.
6. `node_b`는 이를 수신하여 실제 장치를 제어한다.
7. `node_c`도 명령을 수신하여 상태를 반영한다.
8. `node_b`, `node_c`의 상태 발행 결과가 웹 콘솔에 반영된다.

### 6.3 AUTO에서 수동 명령이 무시되는 구조
실제 코드 기준으로 `node_c`는 AUTO 모드에서 수동 `light/window` 명령을 무시한다. 이는 사용자가 실수로 수동 명령을 보내더라도 자동 제어 정책이 우선하도록 하기 위한 장치이다. 반대로 MANUAL 모드일 때만 수동 명령이 허용된다.

## 7. 구현 및 테스트 결과

### 7.1 확인된 통합 흐름
최신 README 기준으로 다음 조합이 확인되었다.

- `node_a + node_c + node_b + web_console` 전체 MQTT 흐름 확인
- `node_b + node_c + web_console` 통합 테스트 확인
- `web_console + node_c` 조합으로 상태 확인 및 수동 제어 가능

### 7.2 시연 가능한 주요 장면
- 센서값 변화에 따라 조명과 창문이 자동으로 동작하는 장면
- 웹 콘솔에서 AUTO / MANUAL 모드를 전환하는 장면
- 웹에서 조명 ON/OFF, 창문 OPEN/CLOSE를 수동으로 제어하는 장면
- `nodeB`, `nodeC` 상태 카드와 heartbeat online 상태가 갱신되는 장면

### 7.3 시스템 강점
- 역할 분리가 명확한 분산형 구조
- MQTT 기반의 높은 확장성
- 중앙 제어 정책 기반의 명확한 제어 로직
- 웹 관제 인터페이스를 통한 시각적 모니터링
- heartbeat와 상태 보고를 포함한 운영형 구조
- WS2812 조명 및 듀얼 서보 창문이라는 실제 동작성이 높은 구현

### 7.4 한계점
- 실제 배선과 전원 안정성에 따라 서보 동작이 민감할 수 있음
- 네트워크 reconnect / DNS 재시도 안정성이 더 보완될 필요가 있음
- 실환경에 맞춘 임계값 보정이 추가로 필요할 수 있음

## 9. 결론

본 프로젝트는 센서 노드, 중앙관리 노드, 액추에이터 노드, 웹 콘솔을 MQTT 기반으로 분리하여 구성한 스마트 하우스 분산 제어 시스템이다.

## 2. 감마(Gamma)용 상세 발표 PPT 제작 프롬프트

아래 프롬프트는 감마(Gamma), Tome, Canva AI, 또는 다른 자동 슬라이드 생성 도구에 넣어 발표 자료를 만들기 위한 상세 프롬프트이다.  
슬라이드 구조, 내용, 스타일, 그림 배치 방향까지 포함하여 최대한 구체적으로 작성하였다.

```text
다음 정보를 바탕으로 대학 팀프로젝트 최종 발표용 PPT를 한국어로 만들어줘.

프로젝트명:
MQTT 기반 스마트 하우스 분산 제어 시스템

발표 대상:
대학교 수업 팀프로젝트 최종 발표

발표 시간:
약 7~10분

발표 목적:
프로젝트의 전체 구조, 기술적 구현, AUTO/MANUAL 제어 로직, 통합 테스트 결과, 프로젝트의 장점과 향후 확장 가능성을 명확하게 전달하는 것

전체 톤앤매너:
- 공학/소프트웨어 프로젝트 발표 스타일
- 지나치게 화려하지 않고 깔끔하고 전문적인 느낌
- 배경은 밝거나 흰색 계열
- 강조색은 파란색, 남색, 회색 계열
- 텍스트는 읽기 쉽게
- 한 슬라이드에 텍스트를 너무 많이 넣지 말 것
- 각 슬라이드는 핵심 메시지가 한눈에 들어오게 구성할 것
- 표, 구성도, 플로우차트, 아이콘을 적절히 활용할 것
- 발표 자료답게 핵심 위주로 정리하되, 너무 단순하지 않게 기술적 설명도 살릴 것

반드시 다음 슬라이드 구조를 따를 것.

1. 표지 슬라이드
- 제목: MQTT 기반 스마트 하우스 분산 제어 시스템
- 부제: HRD 개발자 교육 1차 프로젝트
- 팀원명 또는 팀명 자리 포함
- 한 줄 소개: 센서, 제어기, 액추에이터, 웹 콘솔을 MQTT로 연결한 분산형 스마트 하우스 시스템
- 표지에는 스마트 하우스, 센서, 네트워크, 제어를 연상시키는 심플한 아이콘 또는 배경 요소를 넣을 것

2. 프로젝트 배경 및 목적
- 기존 단일 보드 중심 실습 방식의 한계
- 센서, 제어, 실행, UI를 분리할 필요성
- MQTT 기반 분산 구조를 선택한 이유
- 목표: 환경 데이터를 바탕으로 조명과 창문을 자동/수동 제어하는 시스템 구현
- 이 슬라이드는 문제의식 → 해결 방향 → 프로젝트 목표 구조로 정리할 것

3. 전체 시스템 구성도
- 구성요소: node_a, node_b, node_c, web_console, MQTT Broker
- 계층 구분: 현장 노드 / 중앙 관리 / 사용자 계층
- node_a는 센서 노드
- node_b는 액추에이터 노드
- node_c는 중앙관리 노드
- web_console은 웹 관제 인터페이스
- MQTT Broker는 메시지 허브
- 이 슬라이드에는 전체 시스템 구성도를 크게 넣고, 옆에 각 구성요소의 역할을 짧게 요약할 것

4. 각 노드 역할 설명
- node_a: CDS + DHT 기반 환경 측정, house/env 발행, heartbeat 발행
- node_b: WS2812 조명 제어, 듀얼 서보 창문 제어, 상태/heartbeat 발행
- node_c: AUTO/MANUAL 모드 관리, 임계값 기반 자동 제어, 상태 발행
- web_console: 전체 상태 모니터링, 명령 발행, 로그 표시
- 표나 카드형 레이아웃으로 구성할 것

5. 하드웨어 구성
- node_a: GP15 DHT, GP27 CDS
- node_b: GP14 1번 서보, GP15 2번 서보, GP16 WS2812
- node_c: 중앙 제어용 Pico W
- 최신 구현 특징:
  - WS2812 8픽셀 전체 ON/OFF
  - 듀얼 서보 카세먼트 창문 제어
  - 두 번째 서보는 반대 방향으로 동작
  - secondary -> primary 순으로 시간차를 두고 구동
- 하드웨어 연결을 한눈에 알 수 있는 간단한 표 또는 아이콘 기반 도식 포함

6. MQTT 통신 구조
- 주요 토픽 설명:
  - house/env
  - house/mode
  - house/cmd/light
  - house/cmd/window
  - house/status/nodeB
  - house/status/nodeC
  - house/heartbeat/nodeA
  - house/heartbeat/nodeB
  - house/heartbeat/nodeC
- 누가 발행하고 누가 구독하는지 표 형태로 정리
- env, cmd, status, heartbeat의 의미를 간단히 설명

7. 자동 모드 제어 흐름도
- 흐름:
  1) node_a 센서 측정
  2) house/env 발행
  3) node_c 수신
  4) AUTO 여부 확인
  5) 조도 기반 조명 판단
  6) 온습도 기반 창문 판단
  7) house/cmd/light, house/cmd/window 발행
  8) node_b 장치 제어
  9) node_b / node_c 상태 발행
  10) web_console 상태 갱신
- 조도 기준:
  - light < 280 → ON
  - light > 320 → OFF
- 창문 기준:
  - temp > 28 또는 humidity > 70 → OPEN
  - temp <= 27 그리고 humidity <= 65 → CLOSE
- 이 슬라이드는 플로우차트를 중심으로 구성하고, 오른쪽 또는 아래에 짧은 설명을 배치할 것

8. 수동 모드 제어 흐름도
- 흐름:
  1) 사용자가 web_console 접속
  2) Mode Manual 선택
  3) house/mode = MANUAL 발행
  4) node_c가 MANUAL 전환
  5) 수동 명령 허용
  6) 사용자가 Light / Window 버튼 선택
  7) house/cmd/light 또는 house/cmd/window 발행
  8) node_b는 명령 수신 후 실제 장치 제어
  9) node_c는 명령 수신 후 상태 반영
  10) node_b / node_c 상태 발행
  11) web_console 상태 갱신
- AUTO 모드에서는 수동 명령이 무시된다는 점도 짧게 강조할 것
- 이 슬라이드도 플로우차트 중심으로 구성할 것

9. 웹 콘솔 설명
- Flask + MQTT 기반 관제 웹
- 표시 기능:
  - Environment
  - Controller(node_c)
  - Actuator(node_b)
  - Heartbeat(nodeA, nodeB, nodeC)
  - 최근 MQTT 로그
- 제어 기능:
  - Mode Auto
  - Mode Manual
  - Light On / Off
  - Window Open / Close
- 웹 콘솔 화면 예시가 있다면 이미지 자리 포함
- UI가 프로젝트 시연에서 어떤 역할을 했는지 설명

10. 구현 결과 및 통합 테스트
- 확인된 통합 조합:
  - node_a + node_c + node_b + web_console 전체 MQTT 흐름 확인
  - node_b + node_c + web_console 통합 테스트
  - web_console + node_c만으로도 상태 확인 및 수동 제어 가능
- 실제 시연 흐름 예시:
  - 센서값 변화 → 자동 제어
  - 웹 버튼 클릭 → 수동 제어
  - 상태/heartbeat 반영
- 이 슬라이드는 시연 사진, 로그 캡처, 결과 요약을 배치하는 구조로 만들 것

11. 프로젝트의 장점
- 역할 분리가 명확한 분산형 구조
- MQTT 기반 확장성
- 중앙 정책(node_c) 기반 제어
- 웹 콘솔을 통한 시각적 모니터링
- heartbeat / 상태 snapshot 기반 운영형 구조
- WS2812 + 듀얼 서보를 활용한 실제 동작성 높은 구현
- 장점은 아이콘 + 짧은 문장 형태로 보기 좋게 정리할 것

12. 한계 및 개선 방향
- 네트워크 reconnect / DNS 재시도 안정성 보완 필요
- 실제 배선 및 전원 안정성 보완 필요
- 환경별 임계값 추가 보정 가능
- OLED/LCD 상태 출력 추가 가능
- heartbeat payload 확장 가능
- 추가 센서 / 액추에이터 확장 가능
- 현재 한계와 향후 계획을 균형 있게 정리할 것

13. 결론 및 마무리
- 핵심 요약:
  - sensor, controller, actuator, web UI를 MQTT로 분리한 분산형 스마트 하우스 시스템
  - AUTO / MANUAL 모드 지원
  - WS2812 조명과 듀얼 서보 창문 제어 구현
  - 전체 통합 테스트 완료
- 마지막 한 줄 메시지:
  - 단순 센서 실습을 넘어, 실제 시스템 구조를 갖춘 스마트 하우스 제어 프로젝트
- 깔끔한 마무리 슬라이드로 구성할 것

추가 지시사항:
- 슬라이드마다 제목, 핵심 bullet, 짧은 설명 문장을 자연스럽게 넣을 것
- 텍스트가 너무 빽빽하지 않게 구성할 것
- 전체 슬라이드 수는 약 11~13장 정도로 맞출 것
- 필요하면 표, 아이콘, 흐름도, 계층 다이어그램을 적극적으로 사용할 것

- 한국어로 작성할 것
```

## 3. 감마용 프롬프트 사용 팁

감마 같은 도구에 넣을 때는 아래처럼 사용하면 좋다.

### 사용 방법 1
위의 긴 프롬프트를 그대로 붙여 넣고,  
추가로 아래 한 줄을 마지막에 덧붙인다.

```text
슬라이드별로 시각 자료가 들어갈 위치를 고려해 너무 긴 문단 대신 발표용 bullet과 짧은 설명 중심으로 구성해줘.
```

감마가 너무 텍스트 위주로 뽑으면 아래 문구를 추가한다.

```text
각 슬라이드는 텍스트 40%, 시각 요소 60% 비율로 구성하고, 구성도/플로우차트/표/카드형 요약을 적극적으로 사용해줘.
```

```text
디자인은 화려한 마케팅 스타일보다 공학 발표 스타일로 해주고, 구조도와 데이터 흐름이 명확하게 보이도록 정리해줘.
```

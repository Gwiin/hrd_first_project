## 슬라이드 1. 스마트 환경제어 홈IoT

안녕하세요. 3조 정귀인, 황지용, 김도경입니다. 저희가 진행한 프로젝트는 `스마트 환경제어 홈IoT`이며, 부제는 `MQTT 기반 분산 제어 시스템`입니다. 이번 발표에서는 센서 노드, 중앙 제어기, 액추에이터, 그리고 웹 콘솔을 MQTT로 연결해서 어떤 방식으로 하나의 스마트 홈 시스템을 구현했는지 순서대로 설명드리겠습니다.

## 슬라이드 2. 프로젝트 배경 및 목표

이번 프로젝트는 단순히 센서 값을 읽어보는 실습을 넘어서, 지금까지 학습한 센서 제어, 액추에이터 구동, MQTT 통신, 웹 연동 기술을 실제 시스템 형태로 통합해보는 것이 핵심 목표였습니다. 또 팀 프로젝트인 만큼 각자 역할을 나누어 개발하고, 마지막에 하나의 결과물로 통합하는 협업 과정도 중요하게 생각했습니다. 그래서 저희는 기술적 통합, 학습 내용의 심화, 협업 경험 확보, 그리고 실제로 동작하는 스마트 홈 IoT 시스템 완성을 목표로 프로젝트를 진행했습니다.

## 슬라이드 3. 프로젝트 개요

시스템은 크게 네 가지 요소로 구성됩니다. `node_a`는 조도, 온도, 습도를 측정하는 환경 센서 노드이고, `node_c`는 센서값을 받아 임계값을 기준으로 판단하는 중앙 제어기입니다. `node_b`는 조명과 창문을 실제로 제어하는 액추에이터 노드이며, `web_console`은 전체 상태를 모니터링하고 수동 제어까지 할 수 있는 관제 UI입니다. 흐름으로 보면 `환경 감지`, `중앙 판단`, `장치 제어`, `상태 모니터링`의 구조로 동작합니다.

## 슬라이드 4. 전체 시스템 구성

이 슬라이드는 시스템을 계층 구조로 정리한 화면입니다. 가장 위의 사용자 계층에는 `web_console`이 있고, 사용자는 여기서 현재 상태를 보고 수동 제어를 수행할 수 있습니다. 중앙 관리 계층에는 `MQTT Broker`와 `node_c`가 있어서 메시지를 중계하고 제어 정책을 판단합니다. 그리고 현장 계층에는 `node_a`와 `node_b`가 배치되어 실제 센서를 읽고 장치를 움직입니다. 즉, 각 노드가 직접 서로 강하게 연결된 구조가 아니라 MQTT를 중심으로 느슨하게 연결된 분산 구조라는 점이 핵심입니다.

## 슬라이드 5. 팀원 역할 분배

팀원 역할은 기능 중심으로 분리했습니다. 정귀인은 `node_c`, 웹 콘솔, 전체 통합과 테스트 정리를 맡아서 중앙 제어 로직과 전체 연결 구조를 담당했습니다. 황지용은 `node_b`와 하드웨어를 맡아서 액추에이터 노드 구현, WS2812와 서보 제어, 배선과 모형 제작을 진행했습니다. 김도경은 `node_a`와 문서 작업을 맡아 센서 노드 작업, 연동 지원, PPT 초안 작성과 모형 제작 지원을 담당했습니다. 이렇게 각자 독립적으로 개발한 뒤, MQTT 토픽 규격을 기준으로 부분 통합과 전체 통합을 진행했습니다.

## 슬라이드 6. 하드웨어 및 개발 환경

하드웨어는 `Raspberry Pi Pico 2 W` 3대와 CDS 조도 센서, DHT 온습도 센서, `WS2812 RGB Strip`, `SG90` 서보모터를 사용했습니다. 핀 구성은 `node_a`가 `GP15`에서 DHT를 읽고 `GP27`에서 조도를 측정하며, `node_b`는 `GP14`로 서보를 제어하고 `GP16`으로 WS2812 조명을 제어합니다. 소프트웨어 환경은 MCU 펌웨어에 `C`와 `Pico SDK`, 웹 콘솔에 `Python`과 `Flask`, 통신에는 `Wi-Fi`와 `MQTT`를 사용했습니다. 발표에서 보시면 알 수 있듯이, 센서 입력부터 웹 UI까지 하나의 기술 흐름으로 연결한 점이 특징입니다.

## 슬라이드 7. MQTT 통신

이 프로젝트의 핵심 연결 방식은 MQTT입니다. `house/env`는 `node_a`가 발행하는 환경 데이터 채널이고, `house/cmd/light`, `house/cmd/window`는 `node_c`나 웹 콘솔에서 발행하는 제어 명령 채널입니다. 또 `house/status/nodeB`, `house/status/nodeC`는 상태 동기화를 위한 채널이고, `house/heartbeat/nodeA`, `nodeB`, `nodeC`는 각 노드의 생존 확인용 채널입니다. 정리하면 `node_a`가 환경을 보내고, `node_c`가 판단해서 명령을 내리고, `node_b`가 실행하며, 웹 콘솔은 상태와 heartbeat를 받아 전체 상황을 보여주는 구조입니다.

## 슬라이드 8. 임계값 기반 자동 판단

자동 제어에서는 사람이 직접 개입하지 않아도 센서값에 따라 시스템이 동작합니다. 먼저 `node_a`가 조도, 온도, 습도를 측정해서 `house/env`로 발행하면, `node_c`가 현재 모드가 `AUTO`인지 확인한 뒤 임계값 기준으로 판단합니다. 조도값이 `280`보다 낮으면 조명을 켜고, `320`보다 높으면 끕니다. 또 온도가 `28도`를 넘거나 습도가 `70퍼센트`를 넘으면 창문을 열고, 온도와 습도가 각각 `27도 이하`, `65퍼센트 이하`로 내려오면 창문을 닫습니다. 여기서 조명 ON과 OFF 기준을 다르게 둔 것은 히스테리시스를 적용한 것으로, 경계값 근처에서 장치가 계속 켜졌다 꺼지는 현상을 줄이기 위한 설계입니다.

## 슬라이드 9. 사용자 직접 제어

수동 제어는 `MANUAL` 모드에서만 허용됩니다. 사용자가 웹 콘솔에서 `Mode MANUAL`을 누르면 `house/mode = MANUAL`이 발행되고, `node_c`가 수동 명령을 허용하는 상태로 전환됩니다. 그 다음 사용자가 조명이나 창문 버튼을 누르면 `house/cmd/light` 또는 `house/cmd/window`가 발행되고, `node_b`가 이를 받아 실제 장치를 제어합니다. 반대로 `AUTO` 모드에서는 웹에서 수동 명령을 보내더라도 `node_c`가 정책 일관성을 위해 이를 무시하도록 설계했습니다. 즉 자동 제어와 수동 제어가 서로 충돌하지 않도록 모드 기반으로 명확히 분리했습니다.

## 슬라이드 10. 웹 콘솔

웹 콘솔은 단순한 버튼 화면이 아니라 전체 시스템의 상태를 한 번에 보여주는 관제 화면입니다. `Environment` 영역에서는 조도, 온도, 습도를 실시간으로 확인할 수 있고, `Controller`에서는 `node_c`의 현재 모드와 판단 상태를 볼 수 있습니다. `Actuator`에서는 조명과 창문 상태를 확인할 수 있고, `Heartbeat` 영역에서는 각 노드가 현재 살아 있는지도 확인할 수 있습니다. 또한 최근 MQTT 로그를 함께 보여주기 때문에 명령이 어떻게 오가고 있는지 추적할 수 있고, 여기서 바로 수동 제어도 수행할 수 있습니다.

## 슬라이드 11. 핵심코드 리뷰 1

첫 번째 핵심 코드는 `node_c`의 임계값 기반 자동 제어 로직입니다. `node_c`는 들어온 환경 데이터를 파싱한 뒤 조명과 창문을 각각 따로 판단합니다. 이 구조 덕분에 센서 노드는 단순 측정에 집중하고, 실제 판단 정책은 중앙 제어기 한 곳에서 관리할 수 있습니다. 발표에서 강조하고 싶은 부분은, 제어 기준이 여러 노드에 흩어져 있지 않고 중앙에 모여 있어서 유지보수와 확장이 쉽다는 점입니다.

## 슬라이드 12. 핵심코드 리뷰 2

두 번째 핵심은 `AUTO` 모드에서 수동 명령을 차단하는 로직입니다. 만약 이 기능이 없다면 사용자가 웹에서 버튼을 눌렀을 때 자동 제어 정책과 충돌할 수 있습니다. 저희는 `node_c`가 현재 모드를 먼저 확인한 뒤, `AUTO` 상태에서는 웹이나 MQTT에서 들어온 수동 명령을 무시하도록 처리했습니다. 그래서 사용자가 실수로 버튼을 누르더라도 자동 제어 정책이 깨지지 않도록 했습니다.

## 슬라이드 13. 핵심코드 리뷰 3

세 번째는 중복 명령 방지 로직입니다. 예를 들어 조명이 이미 켜져 있는데 또 `ON` 명령을 계속 보내면 불필요한 MQTT 트래픽이 늘고, 액추에이터도 반복적으로 동작할 수 있습니다. 그래서 현재 상태와 같은 명령은 다시 발행하지 않도록 설계했습니다. 이 부분은 시스템을 단순히 동작하게 만드는 수준을 넘어, 실제 운영 시의 안정성과 효율성을 고려했다는 점에서 의미가 있습니다.

## 슬라이드 14. 핵심코드 리뷰 4

네 번째는 노드 상태 동기화 구조입니다. `node_c`가 명령만 보내고 끝나는 것이 아니라, `node_b`가 실제 반영 결과를 `house/status/nodeB`로 다시 알려주고, `node_c`는 이를 반영해 전체 상태를 정리합니다. 즉 이 프로젝트는 명령 전송만 있는 단방향 구조가 아니라, 실제 결과를 다시 수집해서 상태를 맞춰 가는 구조입니다. 그래서 웹 콘솔에서도 좀 더 신뢰성 있게 현재 상태를 보여줄 수 있습니다.

## 슬라이드 15. 시연 영상

이제 실제 동작 영상을 보시겠습니다. 영상에서는 센서값 변화에 따라 조명과 창문이 자동으로 반응하는 장면, 그리고 웹 콘솔에서 수동으로 조명을 켜고 끄거나 창문을 열고 닫는 장면을 함께 확인하실 수 있습니다. 발표에서는 영상이 재생되는 동안 자동 제어와 수동 제어가 각각 어떤 흐름으로 반영되는지 간단히 짚어주시면 됩니다.

## 슬라이드 16. 테스트 결과

테스트 결과를 보면 전체 통합 흐름이 실제로 확인되었다는 점이 중요합니다. `node_a + node_c + node_b + web_console` 전체 MQTT 흐름을 검증했고, `node_b + node_c + web_console` 부분 통합도 확인했으며, `web_console + node_c`만으로도 상태 확인과 수동 제어가 가능했습니다. 또 `node_c` smoke test 통과, heartbeat 수신과 timeout 경고 확인, status snapshot 발행과 상태 동기화 확인, 중복 명령 방지 로직 확인까지 완료했습니다. 즉 개별 기능 구현에 그치지 않고, 통합 관점에서도 동작을 검증한 프로젝트라고 말씀드릴 수 있습니다.

## 슬라이드 17. 프로젝트 진행 중 어려웠던 점 및 해결 방안

진행 과정에서 크게 두 가지 어려움이 있었습니다. 첫 번째는 협업과 통합 문제였습니다. 처음 팀 프로젝트를 진행하다 보니 GitHub를 활용한 실시간 통합 과정에서 충돌 가능성이 있었고, 이를 해결하기 위해 MCU 역할을 명확히 분리하고 각자 독립 개발 후 최종 통합하는 방식을 선택했습니다. 두 번째는 창문 구동 구조였습니다. 창문의 회전축과 서보모터의 회전축이 달라서 단순 연결로는 자연스러운 개폐 구현이 어려웠는데, 구조를 다시 검토해 싱글 서보 기반 방식으로 재설계하면서 문제를 해결했습니다.

## 슬라이드 18. 결론 및 향후 계획

현재 한계로는 브레드보드 기반 배선의 물리적 불안정성, 센서와 액추에이터 배치에 따른 간섭 가능성, 그리고 환경별 임계값 보정이 아직 고정값이라는 점이 있습니다. 하지만 향후에는 센서 데이터를 저장하고 대시보드로 시각화하거나, 현장 노드에 `OLED`나 `LCD`를 추가하고, 이상 상태 알림 기능과 추가 센서 및 액추에이터를 연결하는 방향으로 확장할 수 있습니다. 즉 현재 구조는 완성된 결과물이면서도, 이후 확장이 가능한 플랫폼 형태로 설계되어 있다는 점이 의미 있습니다.

## 짧게 줄여 말해야 할 때의 핵심 요약

발표 시간이 부족하면 아래 세 문장만 꼭 살리면 된다.

1. 저희 프로젝트는 `node_a`가 측정하고, `node_c`가 판단하고, `node_b`가 실행하며, `web_console`이 이를 모니터링하는 MQTT 기반 분산 스마트 홈 시스템입니다.
2. `AUTO` 모드에서는 임계값과 히스테리시스를 바탕으로 자동 제어하고, `MANUAL` 모드에서는 웹 콘솔을 통해 사용자가 직접 제어할 수 있습니다.
3. 실제 통합 테스트와 시연을 통해 전체 MQTT 흐름, 상태 동기화, heartbeat 기반 운영 확인까지 검증했습니다.
